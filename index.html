Architecture
Open Standard HATP / 0.1

The wire protocol
for AI telemetry

HATP — Hive AI Telemetry Protocol. One canonical event schema. Any provider. Governance baked into every packet. The standard nobody built — until now.

Protocol stack — 5 layers

Layer 5 · Governance
Consent · Residency · Retention
GovernanceBlock is a required field. Protocol rejects events without it. Compliance is enforced at the wire level, not the app level.
Layer 4 · Identity
Signed events · Org attestation
ed25519 signatures over canonical JSON. Node Hub attests provenance. Every event traces to a verified origin.
Layer 3 · Registry
Provider namespace · Model fingerprints
Community-maintained registry of provider IDs and model fingerprints. provider: 'openai' means the same thing everywhere.
Layer 2 · Transport
HTTPS/2 push · Streaming · msgpack
60-second batch flush, up to 500 events. Binary msgpack for high-throughput. SSE streaming for real-time dashboards.
Layer 1 · Schema
HATPEvent · Canonical JSON · Semver
The normative event type. Versioned with backward-compat guarantees. Any compliant implementation produces exactly this.
The standard is open. The network is HIVE.
HATP is MIT-licensed. Any implementation can talk to any HIVE Node. HIVE cannot unilaterally break backward compatibility.

HATPEvent · v0.1 · canonical schema
// Open standard — governed by HATP Working Group
// Reference implementation: HIVE Scout
// Privacy guarantee: content architecturally excluded

interface HATPEvent {

  // Protocol
  hatp_version:  '0.1'
  event_id:      string   // uuid v4
  schema_hash:   string   // sha256 · self-verifying

  // Origin — always hashed
  emitter_id:    string   // rotates monthly
  session_hash:  string   // links req+res · not the user

  // Provider fingerprint
  provider:      AIProvider  // 'openai' | 'anthropic' | ...
  endpoint:      string      // '/v1/chat/completions'
  model_hint:    string      // from response headers · not user

  // Signal — zero content, always
  direction:     'request' | 'response' | 'error'
  payload_bytes: number   // size proxy · never content
  latency_ms:    number
  estimated_tokens: number // derived · not from content

  // Governance — required · no exceptions
  governance: {
    consent_basis:   'org_policy' | 'explicit'
    data_residency:  'AE' | 'EU' | 'US' | 'global'
    retention_days:  number
    regulation_tags: RegulationTag[]
    pii_asserted:    false   // protocol enforced
    content_asserted: false  // protocol enforced
  }

  // Signature — ed25519 over canonical JSON
  signature?:    string
}
@hatp/sdk — drop-in
import { HATProtocol } from '@hatp/sdk'

const hatp = new HATProtocol({
  endpoint: 'https://node.your-org/hatp/ingest',
  governance: {
    data_residency: 'AE',
    regulation_tags: ['UAE_AI_LAW', 'GDPR'],
    retention_days: 90,
  }
})

// Wrap any provider — zero behaviour change
const res = await hatp.wrap(openai.chat.completions.create(…))
// HATPEvent emitted. Governance enforced. Done.

Integrity

TokenPrint scores
nobody can game.

Five independent defence layers protect the leaderboard. Passing one is easy. Passing all five simultaneously, for months, while generating realistic billing patterns — is not.

Defence 01
Velocity anomaly detection

Token rate compared against personal 30-day rolling average. Three-tier response: flag, suspend score update, escalate to integrity review.

Tier 1 · flag: > 20× avg rate
Tier 2 · suspend: > 50× avg rate
Tier 3 · review: billing mismatch
Defence 02
Billing cross-check

Optional: orgs can share provider billing API read-access. HIVE cross-references reported token usage against actual invoice line items. Discrepancy triggers investigation.

Reported: 10B tokens / month
Invoiced: 2B tokens / month
Result: score suspended
Defence 03
Sybil resistance

One Scout per hardware fingerprint. Maximum 3 scouts per person (personal + 2 professional). Org attestation required to earn the professional badge. Unattested scouts score in personal bucket only.

Max scouts / person: 3
Professional badge: org-attested only
Hardware binding: TPM / Secure Enclave
Defence 04
Session bounding

Long-running AI agents (coding assistants, overnight pipelines) are bounded to 10,000 events or 24 hours before generating a new session hash. Prevents agent-dilution attacks where one agent inflates a single user's score.

Max session: 10,000 events
Max duration: 24 hours
Idle timeout: 1 hour
Defence 05
Leaderboard eligibility gates

Public leaderboard requires: minimum 30 days of continuous data, at least 3 distinct models used, org-attested scout, and no active integrity flags. Score is earned — not bought.

Min active days: 30
Min distinct models: 3
Integrity flags: zero tolerance
